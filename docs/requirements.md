# FM OSAKA AI LAB サイト要件定義書

## 0. 【最重要・混同厳禁】サイトには「2種類のコンテンツ」がある

このサイトには、名前が似ていて紛らわしいが **まったく別物の2種類のコンテンツ** がある。
作業指示に出てくる「記事」「解説記事」がどちらを指すかを必ず確定してから着手すること。
**過去に取り違えの事故が発生している（blog記事を作れと誤解し、本来はエピソード解説コンポーネントだった）。**

### A. ブログ記事（＝「最新記事」「blog」）
- 実体: **microCMS の `articles` API に格納されるデータ**（HTML本文）。コードではない
- 表示場所: トップの「最新記事」セクション / `/blog`（一覧）/ `/blog/[slug]`（詳細）
- 作り方: Studio 管理画面（`/studio/new`）で音声→自動生成、または microCMS を直接更新
- エピソードとの紐付け: `episodeSlug` フィールド（`ep01`〜`ep04`）
- **呼ばれ方**: 「ブログ」「最新記事」「blog」「記事をアップして」→ これ

### B. エピソード解説（＝「解説記事」「全4回のAI実験の解説」）
- 実体: **`src/components/episodes/EpisodeNNRequirements.tsx` などハードコードされた React コンポーネント**。microCMS ではなく **コード**
- 表示場所: `/episodes/[slug]`（エピソード詳細ページ）の本文内
- 内容: そのエピソードで作ったツールの **要件定義・仕組み・設計思想** を、非エンジニア／AI初学者向けに作り込んで解説する
- 実装の見本: `Episode01Requirements.tsx`（EP01）。構成は「イントロ→用語メモ→登場するAI→ステップ解説→決めたルール→まとめ」＋解説スライド画像
- ページへの差し込み: `src/app/(public)/episodes/[slug]/page.tsx` が `episode.episodeNumber` で出し分ける
- 一次情報（作り込みのソース）: 要件定義 `../../tools/{ツール名}/requirements.md`（ep01=`fm-radio-blog`、ep02=`reporter-finder`、ep03=`onair-song-checker`）、要望メール `../../mail-logs/`、完成したツール本体（`../../tools/` 等）
- **呼ばれ方**: 「解説記事」「全4回のAI実験の方」「episodes/epNN のやつ」「EP01を参考に」→ これ。**blog ではない**

### 取り違えないためのルール（厳守）
1. **「解説記事」という語は原則 B（エピソード解説コンポーネント）を指す。** B の指示で A（microCMS ブログ）を作らない。逆も同様
2. 少しでも迷ったら、着手前に **「blog（microCMS）の方か、episodes ページ内の解説コンポーネントの方か」を必ず質問して確定する**
3. B を作るときは、必ず要件定義・要望メール・完成ツール本体の3点を読み込んでから作り込む（想像で書かない）

## 1. プロジェクト概要

FM大阪（株式会社エフエム大阪）の番組「iDoBuddy・イドバタニュース」内コーナー **AI Coaching Buddy**（全4回）を紹介し、FM大阪がAIをどのように活用しているかをリスナー・スタッフに分かりやすく届けるサイト。

本サイトは FM大阪のブランド・価値を最優先に設計し、INTENTION の自己顕示ではなく、**「FM OSAKA AI LAB」という取り組み自体が素晴らしいものになる**ことを目指す。

| クライアント | FM大阪（株式会社エフエム大阪） |

| ホスティング | Vercel |
| 運用方針 | **Studio 管理画面経由で更新。記事・画像は microCMS、認証は Supabase Auth に集約。** |

## 2. 目的・ゴール

### 目的
- FM大阪のAI活用（AI Coaching Buddy 全4回）を1つの場所で分かりやすく紹介する
- 放送後すぐにブログ記事・SNS投稿・サムネイルが公開される体験を示す
- リスナーにとって「FM大阪がAIで面白いことをやっている」という印象を与える
- FM大阪スタッフにとって、AIツールの実用性を感じてもらう

### ゴール
- FM大阪公式サイトの延長線上にあるようなテイスト・トーンで統一する
- 第1回〜第4回の企画内容を分かりやすく紹介する
- 生成されたブログ記事・SNS投稿・サムネイルを自然な形で掲載する
- **Studio 管理画面から音声アップロード → 文字起こし → 記事生成 → 公開まで完結させる**
- 軽快に動作し、誰でも迷わず閲覧・運用できるサイトにする

## 7. Supabase 設計

Supabase は **認証** に、microCMS は **記事・画像のコンテンツ管理** に活用する。

### 7.1 Auth（認証）

- Provider: **Email / Password**
- ログインフロー
  1. `/studio/login` でメールアドレス・パスワードを入力
  2. Supabase Auth で認証
  3. `/studio/dashboard` へ遷移
- アクセス制御
  - Middleware で `/studio/*` を保護
  - 未認証ユーザーは `/studio/login` へリダイレクト
  - あらかじめ許可された管理アカウントのみがログイン可能

### 7.2 Storage バケット

| バケット | 用途 |
|---|---|
| `audio` | ラジオ音声ファイル（MP3/WAV/M4A） ※将来利用 |

### 7.3 音声の扱い

- 音声は将来、Supabase Storage `audio` バケットにアップロード予定
- 現時点では管理画面の音声アップロード機能は未実装

## 8. microCMS 設計

### 8.1 `articles` API

| フィールドID | 種類 | 説明 |
|---|---|---|
| title | テキスト | 記事タイトル |
| slug | テキスト | URL用（ユニーク推奨） |
| excerpt | テキストエリア | リード文 |
| body | リッチエディタ | 本文（HTML） |
| thumbnail | テキスト | サムネイル画像URL（microCMS メディアURL） |
| episodeSlug | テキスト | 関連エピソード（`ep01`〜`ep04`） |
| published | 日時 | 公開日時。null の場合は非公開 |

### 8.2 画像の扱い

- **記事関連画像**（サムネイル・記事内画像）は microCMS メディアにアップロード
- **サイト共通画像**（FV 背景、バナーカルーセルなど）も必要に応じて microCMS メディアにアップロードし、URL をコードまたは microCMS フィールドで管理
- microCMS メディアURLを `thumbnail` フィールド、本文中の `img src`、または各コンポーネントの定数に保存
- 管理画面からも手動で画像アップロード可能

## 10. 非機能要件

### 10.1 パフォーマンス
- 初回表示：Lighthouse Performance 70以上を目標
- 画像は Next.js Image で最適化
- 静的ページは可能な限り SSG、動的ページは SSR

### 10.2 SEO
- 各ページにメタタイトル・メタディスクリプションを設定
- OGP 画像対応

### 10.3 セキュリティ
- API キーは環境変数管理
- クライアント側に API キーを露出しない
- **管理画面は Supabase Auth で保護**
- Middleware で未認証アクセスをブロック

### 10.4 アクセシビリティ
- 画像には alt 属性
- キーボード操作可能
- カラーコントラスト確保

```bash
MICROCMS_SERVICE_DOMAIN=your-service-domain
MICROCMS_API_KEY=your-api-key

```

## 12. 運用・更新フロー

1. **Studio 管理画面にログイン**（`/studio/login`）
2. `/studio/new` から音声ファイルをアップロード
3. Whisper で文字起こし → GPT-4o で記事生成
4. 生成結果を確認・編集
5. 「公開」ボタンで microCMS `articles` API に保存
6. 公開済み記事（`published` が null 以外）は自動的にサイトのブログ一覧・詳細に反映
7. 必要に応じて `/studio/articles` から管理・確認
